## 第 8 页｜AI 帮了什么（1 分钟）

### PPT 内容

| 场景 | AI 贡献 |
|------|---------|
| 前端 UI | 快速调样式、节点配色、布局问题 |
| 工作流引擎 | 辅助分叉/汇合执行逻辑 |
| 指令解析 | 补充「替换成」等中文规则 |
| Debug | 根据现象反推根因 |

**协作原则（4 条）**

1. 人定架构，AI 填实现  
2. 小步修改，逐步验证  
3. 要求解释根因，不只要补丁  
4. 登录 / 刷新 / 执行 / 删除 必测  

### 口述

AI 大幅提升了开发效率，但架构和关键决策仍由我把关；小步改、多测试是核心习惯。

## 第 9 页｜踩坑 ①②：执行路径与状态管理（1 分钟）

### PPT 内容

**坑 ① 汇合网关进度不更新**

- 现象：分叉后节点一直「等待中」
- 根因：引擎跳过了汇合节点
- 教训：复杂图结构必须逐节点跑真实路径验证

**坑 ② 刷新/删除状态错乱**

- 刷新进错工作流、删了列表不刷新、切 Tab 丢编辑
- 根因：Pinia 内存态与后端/MySQL 未协同
- 教训：AI 常只改报错点，要测整条状态链

### 口述

这两个问题让我认识到：AI 写的代码能编译通过不等于业务正确，状态管理必须自己画清数据流。

---

## 第 10 页｜踩坑 ③④：指令解析与 UI 细节（1 分钟）

### PPT 内容

**坑 ③「替换成」解析失败**

- 规则库有「替换为」，缺「替换成」
- 策略：**高频指令用规则**，复杂意图再交 LLM

**坑 ④ UI 叠加问题**

- 运行后按钮被状态条遮挡、选中色不对、Step 看不清
- 教训：AI 对 z-index / 对比度 / 动态高度不敏感，需人工走查

**UI 走查四态**：运行前 / 运行中 / 运行后 / 刷新后

### 口述

文档编辑靠规则积累；前端交互细节则要自己过一遍四种状态，不能全靠 AI。
